# Universal Deobfuscator

Мощный инструмент для деобфускации .NET приложений, написанный на C# с использованием библиотеки **dnlib**. Предназначен для анализа и упрощения обфусцированного кода, включая цепочки вызовов методов, математические выражения, control flow obfuscation и wrapper-методы.

## 🚀 Быстрый старт

### Требования
- [.NET 6.0 SDK](https://dotnet.microsoft.com/download) или новее
- [Mono](https://www.mono-project.com/) (для Linux/macOS)

### Установка и запуск

```bash
# Клонируйте репозиторий
git clone <repository-url>
cd UniversalDeobfuscator

# Сборка проекта
dotnet build -c Release

# Запуск
dotnet run -- <путь_к_файлу>.exe
# Или через Mono
mono bin/Release/net6.0/UniversalDeobfuscator.exe <путь_к_файлу>.exe
```

### Пример использования
```bash
dotnet run -- malware.exe --output cleaned.exe --verbose
```

## 🔍 Детальное описание функций

### 1. Анализ цепочек вызовов (`CallChainAnalyzer`)

Самая мощная функция анализатора. Обнаруживает и упрощает цепочки вызовов методов, где один метод просто вызывает другой, создавая многоуровневую структуру для затруднения анализа.

#### Как это работает:
1. **Сканирование**: Поиск всех методов в модуле
2. **Построение графа**: Создание дерева вызовов между методами
3. **Виртуальное выполнение**: Симуляция выполнения IL-кода с отслеживанием:
   - Стека значений
   - Локальных переменных
   - Аргументов методов
   - Условных переходов
4. **Анализ возможностей**: Проверка безопасности инлайна
5. **Оптимизация**: Замена цепочки на прямой вызов финального метода

#### Ключевые ограничения для безопасности:
- Максимальная глубина цепочки: **10 уровней**
- Запрещены методы с обработкой исключений
- Требуется полный баланс стека
- Ограниченная сложность условных переходов

📂 **Исходный код**: [`CallChainAnalyzer.cs`](CallChainAnalyzer.cs)
- `AnalyzeModule()` - главная точка входа
- `SimulateExecution()` - виртуальное выполнение IL
- `CanInline()` - проверка возможности инлайна
- `ApplyOptimizations()` - применение изменений

---

### 2. Математическая оптимизация (`MathOptimizer`)

Упрощает сложные арифметические выражения, часто используемые в обфускаторах для скрытия констант.

#### Примеры преобразований:
```csharp
// До
var x = ((5 * 10) + 25) / 3 - 7;
// После
var x = 23;

// До
if ((a ^ b) ^ b == a) { ... }
// После
if (true) { ... }
```

📂 **Исходный код**: [`MathOptimizer.cs`](MathOptimizer.cs)
- `SimplifyExpressions()` - основной анализатор
- `EvaluateConstantFold()` - вычисление константных выражений
- `OptimizeBitwiseOperations()` - оптимизация битовых операций

---

### 3. Control Flow Unpacking (`ControlFlowUnraveler`)

Восстанавливает оригинальный поток управления из обфусцированных блоков, где логика программы скрыта за сложными переходами и таблицами диспетчеризации.

#### Техники распаковки:
- Анализ switch-конструкций
- Восстановление условий переходов
- Упрощение циклов диспетчеризации
- Удаление мертвого кода

📂 **Исходный код**: [`ControlFlowUnraveler.cs`](ControlFlowUnraveler.cs)
- `UnpackAllMethods()` - сканирование всех методов
- `DetectDispatcherPattern()` - обнаружение диспетчеров
- `ReconstructFlow()` - восстановление потока

---

### 4. Wrapper Inlining (`WrapperInliner`)

Заменяет простые методы-обёртки, которые только передают параметры в другие методы, прямым вызовом целевого метода.

#### Критерии для инлайна:
- Метод содержит только один вызов другого метода
- Отсутствие локальной логики
- Полная передача параметров без изменений
- Совпадение сигнатур

📂 **Исходный код**: [`WrapperInliner.cs`](WrapperInliner.cs)
- `FindWrappers()` - поиск wrapper-методов
- `ValidateWrapper()` - проверка безопасности
- `InlineWrapper()` - замена вызова

---

### 5. Переименование (`Renamer`)

Автоматически переименовывает обфусцированные идентификаторы (классы, методы, поля) в читаемые имена.

#### Стратегии переименования:
- Генерация осмысленных имён на основе контекста
- Префиксы для типов классов/методов/полей
- Сохранение уникальности имён
- Избегание конфликтов с существующими именами

📂 **Исходный код**: [`Renamer.cs`](Renamer.cs)
- `RenameAll()` - главное переименование
- `GenerateClassName()` - генерация имени класса
- `GenerateMethodName()` - генерация имени метода

### Поток выполнения

```
Input Assembly
     ↓
[1] ControlFlowUnraveler    → Восстановление потока управления
     ↓
[2] MathOptimizer           → Упрощение математики
     ↓
[3] CallChainAnalyzer       → Анализ цепочек вызовов
     ↓
[4] WrapperInliner          → Инлайн wrapper-методов
     ↓
[5] Renamer                 → Переименование идентификаторов
     ↓
Cleaned Assembly
```

📂 **Главный оркестратор**: [`UniversalDeobfuscator.cs`](UniversalDeobfuscator.cs)
- `Deobfuscate()` - главная точка входа
- `RunAnalysisPipeline()` - управление последовательностью анализаторов

📂 **Точка входа**: [`Program.cs`](Program.cs)
- `Main()` - обработка аргументов командной строки
- Парсинг параметров `--input`, `--output`, `--verbose`

1. Создайте файл анализатора в корне проекта:
```csharp
// MyCustomAnalyzer.cs
using dnlib.DotNet;

public class MyCustomAnalyzer
{
    public void Analyze(ModuleDefMD module)
    {
        // Ваша логика анализа
    }

```

2. Добавьте вызов в `UniversalDeobfuscator.cs`:
```csharp
new MyCustomAnalyzer().Analyze(module);
```

3. Обновите документацию в README

### Настройка виртуальной машины

Виртуальная машина для выполнения IL находится внутри [`CallChainAnalyzer.cs`](CallChainAnalyzer.cs). Можно расширить поддержку инструкций:

```csharp
case Code.Add:
    var b = stack.Pop();
    var a = stack.Pop();
    stack.Push(a + b);
    break;
```

## ⚠️ Ограничения и предупреждения

### Безопасность
- **Всегда работайте с копией файла**. Инструмент модифицирует сборку.
- Некоторые оптимизации могут изменить поведение программы в редких случаях.
- Методы с обработкой исключений не оптимизируются автоматически.

### Технические ограничения
- Поддерживаются только .NET Framework и .NET Core/5+/6+ сборки
- Не поддерживается нативный код (C++/CLI)
- Сложные динамические вызовы (reflection, dynamic) могут быть не распознаны

### Известные проблемы
- Глубокие цепочки вызовов (>10 уровней) требуют ручной настройки
- Агрессивная обфускация control flow может потребовать дополнительного анализа
- Некоторые упаковщики требуют предварительной распаковки
